# ADR 037 — Documentation site redesign plan

**Status:** Proposed (discussion draft — not yet accepted)
**Date:** 2026-05-17
**Related:** [ADR 028](028-code-first-infra-redesign.md) (the product the docs describe); [ADR 035](035-docs-examples-and-v040-cut-implementation-plan.md) §7.2 (the page-by-page rewrite work item this ADR plans); [ADR 036](036-public-api-naming-critique.md) (the naming the docs commit to); the [redesign tracker](../redesign-status.md) Phase 7
**Phase:** ADR 028 §9.7 (Phase 7), inside §7.2
**Scope of this ADR:** the prose, structure, and assets under `docs/`. CI gates (`mkdocs build --strict`, `tests/docs/test_no_constraint_word.py`), the `mkdocs.yml` skeleton, the brand stylesheets, and the autodoc plumbing in `docs/reference/` are **out of scope** — they are handled by ADR 035 / already landed.

---

## Goal

Rewrite the documentation site so that — from the first visit to the deepest reference page — a reader gets a single, internally consistent story about a single product: **code-first infrastructure for Python.** No constraint-solver leftovers, no half-rewritten tutorials, no aspirational landing page in front of a `0.3.x` how-it-works section.

The output is a site that a Python developer who has never heard of Skaal can read top-to-bottom in under 30 minutes and finish able to answer three questions:

1. *What does this framework actually do?*
2. *How do I run my own app on it in one afternoon?*
3. *When do I reach for it, and when shouldn't I?*

Everything else (deep API reference, advanced patterns, comparison with adjacent tools) is structurally subordinate to those three.

## Why this ADR exists separately from ADR 035

ADR 035 §7.2 says "rewrite every page that mentions constraints" — that's a *task*. This ADR specifies the *target*: tonality, page structure, what survives, what gets cut, what assets to rework. Writing the prose before agreeing on the target is how docs sites end up as a patchwork of voices.

The current site is the cautionary example. `docs/index.md` was already rewritten against the new vocabulary (`Topic`, `Table`, `Blueprint`, `@app.expose`); `docs/how-it-works.md` immediately downstream still describes a Z3 solver picking from a TOML catalog. A reader who clicks "How it works" from the landing page lands on the previous product. ADR 037 makes sure that inconsistency doesn't recur in the rewrite.

## Audience and the first-30-minutes promise

The docs are written for **one** primary reader: a working Python developer who has shipped a backend service before (Flask, FastAPI, Django, Starlette), has touched at least one cloud platform (AWS or GCP) once, and is evaluating whether Skaal saves them from writing a Pulumi program by hand.

Secondary readers, in order of weight:

1. **A returning Skaal user** picking the right primitive or remembering CLI flags. Optimise reference for them: dense, scannable, exhaustive.
2. **A maintainer of an adjacent tool** comparing surfaces. They read the landing page and `comparison.md`.
3. **A contributor** reading ADRs and the reference. Their entry point is GitHub, not the docs site — `CONTRIBUTING.md` and `notes/design/` carry their weight.

We do **not** write for: complete beginners to Python, people who want to learn Pulumi, people who want a generic "infrastructure as code" tutorial. Those readers are politely redirected.

The first-30-minutes promise: landing page → "Get Started" → tutorial 1 → "How it works" → "When to use Skaal". A reader who follows that path reaches an informed adopt/reject decision in one sitting.

## Tonality — the voice of the docs

Pick one voice and hold it across every page. The current site has at least three (clipped/declarative on `index.md`, marketing-prose on `how-it-works.md`, encyclopedic on `cli.md`). The new voice is:

### Direct

- **Lead with the thing.** "`Store[T]` is a typed key-value store." Not "Skaal provides a key-value abstraction in the form of …".
- **Active voice. Present tense.** "Skaal generates a Pulumi program." Not "A Pulumi program is generated by Skaal."
- **Second person, not first.** "You declare `Users(Store[User])`." Not "We then declare …". Never "Let's …".
- **One verb per sentence where you can.** Trim "in order to" → "to". Trim "is able to" → "can". Trim "make use of" → "use".

### Honest

- **Say what doesn't work.** Every page that introduces a feature names at least one thing the feature *doesn't* do. The FAQ already does this well — bring that posture everywhere.
- **No hype words.** Banned: "powerful", "seamless", "blazing-fast", "magical", "simply", "just", "easy", "elegant", "robust". If the framework is good, the code samples show it; the prose stays out of the way.
- **No future-tense roadmap unless gated.** Either the feature exists in `v0.4.0` and the docs describe it, or it doesn't and the docs don't mention it. Forward-looking notes go in an admonition box, never inline prose.

### Show, don't argue

- **Every concept page opens with a runnable code sample under 12 lines.** The prose explains the sample. Not the other way around.
- **Every CLI verb on `cli.md` has a *what you see* paragraph and a *what gets written* paragraph.** Concrete output beats abstract description.
- **No conceptual diagrams without a code anchor.** A diagram is allowed when it's labelling something the reader has just written; standalone "here's an architecture" art reads as filler.

### Concise

- Target length per page: **landing 120 lines, concept pages 200 lines, tutorials 250 lines, reference pages whatever the API demands.** If a page is creeping past these, split it; if it's far under, fold it into its neighbour.
- **No throat-clearing.** Cut "It's worth noting that …", "As mentioned earlier …", "First, let's discuss …". Just say the thing.
- **One link per claim, max.** A paragraph with five inline links is unreadable. Move secondary links to a "Related" footer.

### House style — concrete rules

| Rule | Example |
|------|---------|
| Code identifiers in backticks | `Store[User]`, `@app.expose`, `skaal plan` |
| Proper nouns are bare | AWS, GCP, Pulumi, FastAPI |
| Use **bold** for the first introduction of a term, then plain text | "A **Blueprint** is the environment-independent shape of your app …" |
| Headings are sentence case, not title case | "How it works", not "How It Works" |
| One H1 per page, used once at the top | (Material requires this) |
| American English spelling | "behavior", not "behaviour" — already in `CLAUDE.md` |
| Bullets carry exactly one idea | Split overloaded bullets into two |
| Tables for comparisons; lists for steps; prose for explanations | Don't mix |

## What gets removed

Cut without replacement:

- **Every reference to the constraint vocabulary in prose.** `Latency`, `Durability`, `AccessPattern`, `Throughput`, `Constraint`, `@app.handler`, `@app.scale`, `@app.shared`. The `tests/docs/test_no_constraint_word.py` gate already hard-fails on twelve of these and `xfail`s the remaining four; this rewrite flips the four `xfail`s green.
- **The Z3 / SMT solver narrative.** Every "the solver picks …" sentence. Replaced by "the binding layer looks up …".
- **TOML catalog explanations.** Catalogs are deleted in Phase 1; pages still referring to `catalogs/local.toml`, the `[catalog]` block, or `--catalog <path>` flags either get rewritten against `skaal.toml` or get deleted entirely.
- **`docs/faq.md` "Solver behavior" section** — three questions about `UnsatisfiableConstraintsError`, solver speed, and pinning. All three premises are gone.
- **`docs/faq.md` "Why solve?" / "Is the solver overkill for small apps?"** — same.
- **`docs/comparison.md` references to "constraint-driven selection"** — replace with "code-first inference" or delete the bullet.
- **`docs/getting-started.md` "Core Command Loop" snippet** that runs `skaal plan examples.counter:app --target local --catalog catalogs/local.toml`. Replace with the new `--env` flow.
- **`docs/tutorials/planning-and-deployment.md`** in its current form — it teaches the catalog model. Rewrite from scratch against `skaal plan` / `skaal build` / `skaal deploy` driven by `skaal.toml`.
- **`how-it-works.md`'s four-card flow grid** as currently written. The cards are correct *shape* (four steps, illustration + prose) but every label is wrong. Re-author against the **Declare → Infer → Bind → Generate → Deploy** five-step pipeline.
- **The `Examples` page's table-dashboard / mesh-counter / dash-app rows.** ADR 035 §7.1 already deleted those examples on disk; the doc still lists them.

Cut and rewrite as something narrower:

- **`docs/platform-features.md`** — currently a survey of every primitive the framework offers, weighted toward catalog/solver framing. Replace with **"What you can build"** (six concrete app shapes the reader recognises: HTTP API, scheduled job, fan-out worker, blob upload, relational app, mounted FastAPI service).
- **`docs/cli-configuration.md`** — currently mixes `skaal.toml`, `pyproject.toml` `[tool.skaal]`, and catalog overlay docs. Drop the catalog overlay section; reframe as **"Configuring your environments"** (`skaal.toml` only, with `[tool.skaal]` and env vars as escape hatches).
- **`docs/http.md`** — currently OK in shape (it describes the ASGI mount model) but uses the deleted `app.handler` decorator in examples. Rewrite examples against `@app.expose` + mounted FastAPI; cut the section on auto-route generation that no longer exists.

## What gets added

New top-level pages:

- **`docs/whats-new.md`** — already required by ADR 035 §7.4. A *forward-looking* one-pager: what `v0.4.0` ships, what's deferred to `v0.4.x`, where the upgrade path from `0.3.x` lives (which is: there isn't one — pin `0.3.1` if you need the constraint product). Linked from the landing page status callout.
- **`docs/when-to-use.md`** — a short, honest "should you adopt this?" page. Three columns: *use Skaal*, *use plain Pulumi*, *use a managed PaaS*. The framework's adoption rate is gated more by reader self-selection than by missing features; this page does the selection for them and saves everyone time. Linked from the landing page header.
- **`docs/concepts.md`** — a single-page glossary of the eight terms the docs commit to: `App`, `Module`, `Store` / `Table` / `BlobStore` / `Topic`, `Blueprint`, `Plan`, `Environment`, `Backend`, `Binding`, `Lock`. Three-sentence entries with a code snippet each. This is the page maintainers link to from every other page that introduces a term in passing.

New material inside existing pages:

- **Each tutorial gets a "What you'll be able to do after" header** — a five-bullet summary of capabilities, not concepts. Readers self-select faster.
- **Each tutorial gets a "What this doesn't cover" footer** — explicit pointer to the next tutorial or guide that does cover it.
- **`docs/how-it-works.md` gets a "Pipeline" inline diagram** (the existing `source-to-resource-arrow.svg` is close in shape; it gets re-labelled).
- **`docs/cli.md`'s every verb gets a "Common failures" block** — the two or three things that go wrong in practice, with the exact error text the user sees. The current page is exhaustive on flags and silent on failure modes.

## Sitemap — the new structure

Current nav has three top-level groups (Tutorials, Guides, Reference) plus Home and Get Started. The new nav keeps that shape but reorders and trims:

```yaml
nav:
  - Home: index.md
  - Get started: getting-started.md
  - Concepts: concepts.md                       # NEW — glossary anchor
  - Tutorials:
      - Overview: tutorials/index.md
      - 1. Your first app: tutorials/first-app.md
      - 2. Adding HTTP routes: tutorials/http-api.md
      - 3. Planning and deploying: tutorials/planning-and-deployment.md
      - 4. Relational data: tutorials/relational-and-migrations.md
      - 5. Files and streaming: tutorials/files-and-streaming.md
  - Guides:
      - How it works: how-it-works.md
      - When to use Skaal: when-to-use.md       # NEW
      - What you can build: platform-features.md # RENAMED (was Platform Features)
      - HTTP integration: http.md
      - FAQ: faq.md
  - Reference:
      - CLI:
          - Commands: cli.md
          - Configuration: cli-configuration.md
      - Python API:
          - Overview: reference/python-api.md
          - Core and decorators: reference/python-api-core.md
          - Data surfaces: reference/python-api-data.md
          - Scheduling and components: reference/python-api-components.md
          - Types and policies: reference/python-api-types.md
          - CLI-parity API: reference/python-api-cli-parity.md
  - Comparison: comparison.md                    # MOVED to top-level — high-value evaluator page
  - What's new: whats-new.md                     # NEW
  - Examples: examples.md
```

Rationale for the reordering:

- **"Concepts" moves up to top-level.** Today it's buried inside the Python API reference; readers don't reach it before they need it.
- **"Comparison" moves out of Guides to top-level.** This is one of the highest-leverage pages for evaluators; making it one click from the homepage matters more than category-purity.
- **"What you can build" sits between How-it-works and HTTP integration.** That ordering matches reader intent: *what does it do conceptually → what kinds of apps does it run → how do I wire HTTP to it.*
- **CLI splits into Commands + Configuration as a nested reference group.** The current "CLI" and "CLI Configuration" being siblings of FAQ on the same level reads as four equally-weighted topics; they're not.
- **`tutorials/index.md` stays.** It's the only nav location where a numbered progression makes sense.

## Sequencing

The rewrite is naturally ordered by reader-encounter depth — fix what readers see first, last. Each step ends with `mkdocs build --strict` green and `tests/docs/test_no_constraint_word.py` showing one more `xfail`-flipped-to-pass.

1. **Concepts and structure first.** Land `concepts.md`, `when-to-use.md`, the `mkdocs.yml` nav reshuffle, and the three asset reworks (`primitive-tokens.svg`, `binding-card.svg`, `pipeline-arrow.svg`). Nothing references them yet, but the surface is in place.
2. **Rewrite `how-it-works.md`.** It consumes the new SVGs and is the highest-traffic explanation page. After this, the landing page is no longer lying.
3. **Rewrite the tutorial track end-to-end as one PR.** Tutorials must be internally consistent — splitting them across PRs produces voice drift.
4. **Rewrite `cli.md` and `cli-configuration.md`.** Reference-density work; lower drift risk.
5. **Rewrite `faq.md`, `comparison.md`, `platform-features.md` → "What you can build".** Lower-traffic pages; safer to land last.
6. **`whats-new.md` and final landing-page polish.** Landed alongside the `v0.4.0` cut.

Each step is independently CI-gated. The site is shippable after step 1; each subsequent step monotonically improves it.

## Acceptance criteria

- `tests/docs/test_no_constraint_word.py` — all four remaining `xfail strict=True` tokens (`constraint`, `Constraint`, `Latency`, `VectorStore`) flip to `XPASS` (i.e. zero hits in `docs/`).
- `mkdocs build --strict` is green at every step.
- Every page in the new sitemap exists and links from the nav.
- The four reworked SVGs (`primitive-tokens.svg`, `binding-card.svg`, `pipeline-arrow.svg`, `pulumi-output.svg`) load without 404s.
- A maintainer reads the docs top-to-bottom and confirms: (a) one voice, (b) no `0.3.x` vocabulary, (c) the first-30-minutes promise holds for a new reader.
- ADR 035 §7.2 checkbox flips to ticked.

## Open questions

1. **Land before or after ADR 036?** If ADR 036 (the naming refresh) is accepted, the docs should be written against the new names — meaning ADR 036 lands first, this ADR's rewrite second. If ADR 036 is deferred to `v0.5.x`, this rewrite proceeds against today's names and accepts that a second pass is needed later. The cheaper sequencing is "036 first, 037 second"; the conservative sequencing is "037 against current names, accept rework cost later". Maintainer call.
2. **Add a "Cookbook" or fold recipes into "What you can build"?** A separate cookbook page can grow without bound; folding into `platform-features.md`/`when-to-use.md` keeps the nav lean but caps content. Default: fold; revisit when there are ≥ 10 recipes worth writing.
3. **Keep `examples.md` as a doc page, or replace with a GitHub-only `examples/README.md`?** The doc page duplicates the on-disk README and goes stale. Default: keep, but auto-generate the table from `examples/*/README.md` front-matter.
4. **Migration guide from `0.3.x` — yes or no?** ADR 028 commits to no backwards-compatibility shims. A migration guide implies a migration path. Default: no guide; one Q in the FAQ ("pin `0.3.1`") is the whole story.
5. **Should `concepts.md` ship before or after ADR 036?** If 036 lands, the glossary is authoritative and every other page links to it. If 036 is deferred, the glossary fixes today's vocabulary in place, which is exactly what ADR 036 is trying to avoid. Sequencing answer falls out of Q1.

## Decision

Not yet decided. This ADR sets the target for ADR 035 §7.2; the next step is the maintainer accepting / amending the sitemap, the tonality rules, and the asset disposition table before the page-by-page rewrite PRs begin.
